```
00-overview  →  01-product  →  02-specs  →  03-design  →  04-tasks  →  05-conventions
   (지도)         (목적)         (재료)        (설계)        (일정)         (규칙)
```

## 왜 이렇게 나누는가 — 관심사 분리 원칙

코드에서 관심사 분리(Separation of Concerns)가 모듈을 독립적으로 만들듯,
문서도 **변경 이유가 다른 정보는 다른 파일에 둔다.**

| 변경 이유 | 해당 문서 |
|-----------|-----------|
| 타깃·목표가 바뀔 때 | `01-product.md` |
| 기술 스택·버전이 바뀔 때 | `02-specs.md` |
| 설계 결정이 바뀔 때 | `03-design.md` |
| 일정·우선순위가 바뀔 때 | `04-tasks.md` |
| 팀 규칙이 바뀔 때 | `05-conventions.md` |

한 파일에 모든 내용을 몰아넣으면:
- 작은 변경에도 전체 문서를 뒤져야 한다.
- 어떤 결정이 어디서 왔는지 추적이 어렵다.
- 협업 시 충돌 가능성이 높아진다.

분리하면:
- 변경 범위가 명확해진다.
- 새 팀원이 필요한 문서만 골라 읽을 수 있다.
- Claude가 작업 전 정확히 어디를 읽어야 할지 알 수 있다.
